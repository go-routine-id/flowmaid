Overview

flowmaid is a diagram engine that speaks Mermaid syntax and is written in pure Rust with zero dependencies — no C libraries, no JavaScript runtime, nothing beyond the Rust standard library. Feed it Mermaid-like text, get back an SVG string or plain positioned geometry you can paint with any UI toolkit.

📦 crates.io/crates/flowmaid 📚 docs.rs/flowmaid ⭐ go-routine-id/flowmaid ▶ Live playground

Highlights

  • 8 diagram types — flowchart, ER, class, sequence, state, pie, mindmap, user journey.
  • Mermaid-grade layout — a faithful dagre-style pipeline: crossing minimization, Brandes-Köpf coordinates, subgraph clusters with border walls, edge labels that reserve their own space, smooth curveBasis splines.
  • Interactive by design — a scene/route API exposes every node and edge as plain geometry, so hosts can implement drag & drop and re-route edges without re-running layout.
  • Runs anywhere Rust runs — native CLI, desktop apps, or the browser via a ~380 KB WebAssembly bundle (that is what powers the playground).
  • Line-numbered errors — parse failures tell you the offending line.

Installation

Requires Rust 1.75 or newer. flowmaid has no other requirements.

As a library

cargo add flowmaid

or in Cargo.toml:

[dependencies]
flowmaid = "0.17"

As a command-line tool

cargo install flowmaid

This installs a flowmaid binary that converts .mmd files to SVG.

Quick start — CLI

# file in, file out
flowmaid diagram.mmd -o diagram.svg

# or classic UNIX pipes
cat diagram.mmd | flowmaid > diagram.svg

Try it with this diagram.mmd:

flowchart TD
    A([Start]) --> B[Read input]
    B --> C{Valid?}
    C -->|yes| D[Process data]
    C -->|no| E[Show error]
    E --> B
    D ==> F((Done))

flowmaid --help lists the options.

Quick start — Rust library

The one-call API renders any supported source straight to an SVG string:

fn main() {
    let svg = flowmaid::render_svg("flowchart TD\nA --> B{OK?}").unwrap();
    std::fs::write("out.svg", svg).unwrap();
}

For more control, parse first and match on the document type:

use flowmaid::{parser, model::Document};

match parser::parse_document(src) {
    Ok(Document::Flowchart(g)) | Ok(Document::State(g)) => {
        let scene = flowmaid::scene::scene(&g);   // positioned geometry
        let svg   = flowmaid::scene::to_svg(&scene);
    }
    Ok(Document::Er(d))       => { /* flowmaid::er::scene(&d) … */ }
    Ok(Document::Sequence(d)) => { /* flowmaid::seq::scene(&d) … */ }
    Ok(other)                 => { /* class, pie, mindmap, journey … */ }
    Err(e) => eprintln!("parse error: {e}"),      // line-numbered
}

Labels support Mermaid's inline markup: <br/> line breaks and <b>/<i> styling render as real bold/italic text.

Interactive apps (drag & drop)

Every diagram type exposes its geometry as a scene: node centres, sizes, shapes, edge curves, cluster boxes — plain data, no canvas assumed. The flow for an editor with draggable nodes:

use flowmaid::scene;

let s0 = scene::scene(&graph);                 // 1. automatic layout
let mut pos: Vec<(f64, f64)> =                 // 2. host owns positions
    s0.nodes.iter().map(|n| (n.x, n.y)).collect();

// … user drags node i, host updates pos[i] …

let s1 = scene::route_partial(                 // 3. LOCAL re-route:
    &graph, &pos, &s0, &auto_pos);             //    untouched edges keep
                                               //    engine-quality routing
let svg = scene::to_svg(&s1);                  // 4. export any state
  • scene() — full automatic layout (what the CLI and playground use).
  • route(g, &pos) — re-route every edge for arbitrary positions.
  • route_partial(g, &pos, &base, &base_pos) — re-route only edges touching moved nodes; the rest keep their channel-routed geometry.
  • The same triple exists for ER (er::), class (class::) and mindmap (mindmap::) diagrams; sequence, pie and journey are static.

WebAssembly

The flowmaid-web repository compiles the engine to a ~380 KB wasm bundle with tiny JS bindings — the playground is built on exactly this:

import init, { render_svg } from "./pkg/flowmaid_web.js";

await init();
document.body.innerHTML =
    render_svg("flowchart TD\nA([Start]) --> B{OK?}");

Additional bindings power interactivity: auto_positions, node_sizes, node_keys, and render_routed(src, positions) for drag-and-drop, plus engine_version().

Building the bundle yourself

git clone https://github.com/go-routine-id/flowmaid-web
cd flowmaid-web
wasm-pack build --target web --release   # emits pkg/
python3 -m http.server                   # open http://localhost:8000

How layout works

Flowcharts go through a Sugiyama pipeline modelled on dagre — the same algorithm family mermaid.js uses:

  1. DFS marks back-edges so cycles cannot break the layering.
  2. Longest-path layering assigns nodes to ranks.
  3. Long edges split into per-rank dummy chains that reserve routing channels; a labelled edge sizes a dummy to its label so labels claim real space.
  4. Ordering cuts crossings with weighted-median sweeps plus an adjacent-swap transpose, keeping the best round. Subgraph members stay contiguous, bracketed by invisible border walls.
  5. Brandes-Köpf coordinate assignment straightens long edges and centres fan-in/outs; edges then flow through their channels as curveBasis splines.

Sequence, pie, mindmap and journey diagrams use their own dedicated layouts.

Ecosystem

flowmaid The engine — parser, layout, SVG renderer, interactive scene API (this documentation).

flowmaid-web WebAssembly bindings + the live playground.
